# takumi

Claude Code で、あなたの開発チームをまるごと 1 つのコマンドに。

```
/takumi 管理画面に CSV エクスポート機能を追加
```

このひとことで、要件のヒアリングから設計・実装・テスト作成・レビューまで、シニアエンジニア数名ぶんの仕事が走り出します。

---

## こんなお悩み、ありませんか?

AI コーディング支援ツール (Claude Code、Cursor、Cline、Aider など、こうしたツールはまとめて「ハーネス」と呼ばれます) を使っていると、こんな壁にぶつかることがあります。

- AI に実装を頼んでも、テストが甘くて本番で壊れる
- セキュリティやパフォーマンスの観点を、ひとりではすべて見きれない
- 大きな機能を任せると、途中で話が脱線したり、コンテキストを見失ったりする
- 長時間の作業を任せたいのに、席を外すと止まってしまう
- 覚えるコマンドやスキルが多すぎて、どれをいつ使えばよいのかわからない
- AI が書いたコードを、誰が・何をもってレビューすればよいのか迷う

takumi は、これらの課題を Claude Code 標準の仕組みの上に積み上げた 1 枚のスキルで解決することを目指しています。

---

## takumi でできること (6 つの特徴)

### 1. 覚えるコマンドは `/takumi` ひとつだけ

AI 開発ツールはコマンドが増える傾向にあります。Claude Code にも `/plan`、`/review`、`/security-review` などが標準で用意されていますし、Cursor には Composer、Cline には Plan Mode / Act Mode と、ハーネスごとに異なる概念があります。

takumi は逆の発想です。**入り口を `/takumi` 1 つに絞り、自然な日本語を投げるだけで、中で自動的にモード分岐します。**

| こう話しかけると | 中ではこう動きます |
|---|---|
| `/takumi 商品一覧にソートを追加` | 新機能を実装するモード |
| `/takumi security 見て` | セキュリティ観点で診断するモード |
| `/takumi リリース前に全般見て` | 全方位で総点検するモード |
| `/takumi 今なに動いてる?` | 現在の状態を確認 |
| `/takumi 続きから` | 前回の途中から再開 |
| `/takumi 止めて` | 一時停止 |

命令を丸暗記する必要はありません。「こう言えばこう動くかな」という直感でおおむね通じます。迷ったときだけ、takumi が 1 問だけ聞き返します。

### 2. 「自分が今持っていないレビュー観点」が、その場で手に入る

一人前のエンジニアになるには、実装力だけでなく、コードを多角的に読む「レビュー観点」を身につける必要があります。たとえば次のようなものです。

- **セキュリティレビュー** — SQL インジェクション、権限昇格、CSRF といった攻撃経路を見抜く観点
- **パフォーマンスレビュー** — N+1 クエリ (1 件ごとに DB を叩いてしまう問題)、不要な再レンダリング、メモリリーク
- **アクセシビリティレビュー** — WCAG (Web Content Accessibility Guidelines) への準拠、スクリーンリーダー対応、キーボードだけで操作できるか
- **リファクタリングレビュー** — 責務分離、依存の向き、命名の適切さ
- **テスト戦略レビュー** — 何をどの層で守るのか、不要なテストはないか

これらをすべて高い水準で維持するのは、10 年選手でも難しいことです。

takumi には、これらの観点に対応する専門スキルが内蔵されています。`/takumi security 見て` と話しかけるだけで、該当するレビューが自動で走ります。

- **strict-refactoring スキル** — リファクタリングと設計の指針 (後述)
- **verify スキル** — テスト戦略 (後述)
- **design スキル** — UI の情報設計 (サイトマップ) ・スタイルガイド・ワイヤーフレーム生成

つまり、**これまでチームに居なかった専門家が、会話一つで一時的に加わるような体験**です。まだ経験が浅くても、シニアが隣でレビューしているのと同じような観点で自分のコードを見てもらえます。

### 3. テストを、ここ 50 年で一番マシな方法で書いてくれる

少し丁寧にご説明させてください。ソフトウェアテストは歴史的に、次のような流れをたどってきました。

**昔ながらのテスト (example-based unit test)**
「入力 A を渡したら B が返ること」をテストとして 1 つずつ書く方式です。人間が思いついた入力しか試せないので、境界値や組み合わせの漏れが起きやすいという弱点があります。

**カバレッジ至上主義**
「テストで実行されたコード行の割合」を 80% などの目標で管理する方式です。しかし、行が実行されていても、肝心な assert (検証) が甘ければ何も守れていません。研究によると、カバレッジ 80% を達成していても、バグの 30% 以上を見逃すことが知られています。

**ここ 50 年で研究者たちが提案してきた、本当にいい指標たち**

- **Mutation Testing (ミューテーションテスト)** — テストそのものではなく、本番コードのほうをランダムに書き換え (例: `>` を `>=` に、`true` を `false` に、`+` を `-` に) て、その「人工的に植えたバグ」をテストが検知できるかを測る手法です。検知できなかった「変異」の割合が低いほど、テストが鋭いことを意味します。
- **Property-Based Testing (プロパティベーステスト、PBT)** — 「ケースを 1 つずつ」ではなく、「どんな入力でも成り立つ性質 (プロパティ)」を書いて、AI やライブラリが入力を 1 万回ランダム生成して試す手法です。QuickCheck という最初の実装が 1999 年に登場してから 25 年以上経ちます。
- **Metamorphic Testing (メタモルフィックテスト)** — 正解を直接書けない領域 (画像生成、機械学習、LLM の出力など) で、「入力をこう変換したら、出力もこう変換されるはず」という**関係**で検証する手法です。たとえば「画像を反転させたら、顔検出の座標も反転するはず」のような性質です。
- **Model-Based Testing (モデルベーステスト)** — 「状態機械」をテスト側に用意し、そこをランダムに歩き回る操作列を自動生成して試す手法です。UI の複雑なステート管理 (ログイン→検索→カート→決済→…) のテストに特に有効です。

**これらが普及してこなかった理由はひとつで、人間が使うには難しすぎたからです。** 「どんな入力でも成り立つ性質」を言語化することも、正解のない出力に「関係」を定義することも、状態機械を正しく設計することも、それぞれかなりの専門性を要求します。

AI によってその障壁がほぼ消えました。takumi は仕様 (後述する AC-ID) と関数シグネチャから、これらのテストを自動生成します。ユーザーがやることは **「何を守りたいか」を日本語で伝えること**、それだけです。

### 4. リファクタリングに、人類が長年積み上げてきた知恵を込めてあります

リファクタリングや設計判断は「正解がない」と言われがちですが、実際には**コミュニティが長年かけて発見してきた、より良いパターン**というものが存在します。

- **オブジェクト指向プログラミング (OOP)** — 1970 年代から磨かれてきたカプセル化・責務分離の考え方
- **関数型プログラミング (FP)** — 副作用を分離して純粋関数を中心に置く考え方
- **ドメイン駆動設計 (DDD)** — 2003 年以降広まった、業務領域 (ドメイン) を中心に据える設計手法
- **CQRS (Command Query Responsibility Segregation)** — 「状態を変える操作 (Command)」と「状態を読む操作 (Query / ReadModel)」を分離する考え方
- **Pending Object Pattern** — UI の中間状態を明示的にオブジェクト化して、入力を validate してからコミットする手法

takumi の **strict-refactoring スキル**は、これらの蓄積をチェックリストとして持っています。しかもコンテキストに応じて強度を可変にします。

- **新規のドメインコード** → 厳密に適用 (Command / Pure / ReadModel に分類)
- **既存のレガシーコードを少しだけ触る場合** → 緩めに適用 (警告レベル)
- **外部 API との接続層** → さらに緩めに (DTO 層として割り切る)

「教科書には書いてあるけれど、現場では守れない」というパターンを、場所と状況に応じて柔軟に運用します。

### 5. 計画ファイル化 — 放置してもずっと動き続けます

大きな機能を AI に頼むと、最初は調子よく進むのに、途中から話が脱線したり、文脈を見失ったりする経験、ありませんか?

takumi は会話を始めてすぐ、やることを `.takumi/plans/{機能名}.md` というテキストファイルに書き出します。このファイルには次のような情報が含まれます。

- 作業の塊 (Wave 1、Wave 2、Wave 3 …) とその実行順
- 各タスクの対象ファイル、変更内容、検証手順
- 担当ロール (実装係、レビュー係、調査係)
- 合格ゲート (mutation score 何%以上、レイアウト不変条件がいくつ成立しているか、など)

このファイル化には 3 つの利点があります。

**A. いつでも再開できます**
セッションが切れようと、PC を閉じようと、`/takumi 続きから` のひとことで続きに戻れます。AI が持っていた作業記憶は計画ファイルに吐き出されているので、再読み込みするだけで続きが走ります。

**B. 長時間の自動運転に強いです**
100 個以上のタスクに膨らんだ大きな計画でも、Wave ごとに検証ゲートを通しながら進むため、迷子になりません。寝る前に `/takumi リリース前の総点検` と投げて、朝コーヒーを淹れるころには 50 件の問題が片付いていた、ということが現実に起こります。

**C. 人間がレビューできます**
AI がコードを書き始める**前**に、計画のテキストを人間が読んでレビューできます。「Wave 3 の認可タスクは粒度が粗い、2 つに分けて」のようなフィードバックが自然に返せます。

また、大きな総点検では**自己増殖型計画**になります。職人 (実装担当 AI) が実装中に見つけた別の問題は `discovered-{id}.md` に記録され、Wave 完了時に棟梁 (統括 AI) が計画に追記します。**計画そのものが、作業しながら成長していきます。** 「始める前にすべてを予測する必要がない」のが強みです。

### 6. 4 つのロールを、それぞれ得意分野で使い分けます

takumi は内部で、役割に応じて 4 種類の AI エージェントを使い分けます。

| ロール | モデル | 担当 |
|---|---|---|
| 棟梁 (とうりょう) | Claude Opus | 全体設計、ヒアリング、計画作成 |
| 軍師 (ぐんし) | OpenAI GPT-5 | 別視点からの敵対的レビュー |
| 職人 (しょくにん) | Claude Sonnet | 実装、テスト作成 |
| 斥候 (せっこう) | Claude Haiku | コードベースの高速探索 |

特に大事なのが **軍師** です。実装したのと同じ AI モデルにレビューを頼むと、同じクセで同じ見落としをしがちです。別系統のモデル (GPT-5) にクロスレビューを頼むことで、「ある AI には見えなかった問題」が見えるようになります。これは複数の視点でコードを守る、とても大事な仕組みです。

## 実際に話しかけてみましょう

インストール後、プロジェクトのルートで `/takumi` とだけ送ると、プロジェクトの種別 (UI を含むか、バックエンドのみかなど) を 1 問だけ聞かれます。そこから先は自然文で構いません。以下、場面ごとの例です。

### 新機能を作りたいとき

```
/takumi Stripe 決済を追加。単発購入のみ、サブスクリプションはなし
/takumi ユーザー削除に GDPR 対応の論理削除と cascade を入れる
/takumi 記事投稿画面に 10 秒ごとの下書き自動保存を追加
/takumi 管理画面の商品一覧にソート・フィルター・CSV エクスポート
/takumi Webhook エンドポイントに署名検証を追加
```

### 既存のバグを直したいとき

```
/takumi ログイン後の遷移が遅い、原因を調べて直して
/takumi N+1 が出てそうな API を探して修正
/takumi 検索が壊れている。特定の入力で 500 になる
/takumi 通知メールの文面を刷新したい。既存の雰囲気は崩さずに段階的に
```

### 観点別に診断したいとき (コードは触らずに、まず調べる)

```
/takumi security 見て
/takumi 認可ロジック、権限昇格の抜けがないか調べて
/takumi パフォーマンスが心配。遅い top 5 の endpoint を洗い出して
/takumi a11y 調べて。WCAG AA で落ちそうなところ
/takumi 並行編集が怪しい。レースコンディションの可能性ある?
```

観点診断モードではまず**発見**だけを行い、`.takumi/sprints/{日付}/discoveries.md` に結果を書き出します。そのうえで「提案 1 と 3 だけ直して」と指示すれば、修正計画の生成に進みます。**いきなりコードを触らないので、様子見に最適です。**

### リリース前に総点検したいとき

```
/takumi リリース前に全般見て
/takumi 総点検。security、パフォーマンス、a11y、dx 全部
/takumi 来週リリースなのでリリースブロッカーだけ洗い出して
```

### リファクタリングや設計相談

```
/takumi この UserService、責務が多すぎる気がする
/takumi 状態管理が複雑化してきた。整理できる?
/takumi 既存の checkout 画面、リファクタ観点で見直して
```

### テストを強化したいとき

```
/takumi PricingCalculator に PBT を追加して mutation score を 80% 以上にして
/takumi この feature のテスト戦略を提案して
/takumi 画像リサイズ関数、正解がないので metamorphic な性質で守って
```

### デザインを含む新機能 (UI 設計から)

```
/takumi SaaS の pricing page を作って。参考: Linear、Vercel。トーン: 落ち着いた monochrome
/takumi 管理画面のダッシュボードをダークモード対応に刷新
```

### 運用コマンド

```
/takumi 今なに動いてる?
/takumi 続きから
/takumi 止めて
/takumi auth の loop だけ止めて、mutation loop は続けて
```

## 他のハーネスとの違い

Cursor、Cline、Aider、Continue.dev など、モードや計画の概念自体はすでに各ハーネスに存在します。takumi の立ち位置を明確にするために比較しておきます。

| | Cursor Composer | Cline Plan/Act | Aider /architect | **takumi** |
|---|---|---|---|---|
| モード選択 | ユーザーが選ぶ | ユーザーが切替 | ユーザーがコマンド指定 | **自然文から自動推定** |
| 計画の永続化 | セッション内 | セッション内 | セッション内 | **ファイルとしてコミット可能** |
| セッション跨ぎの再開 | 限定的 | 限定的 | 限定的 | **`続きから` 1 語で復元** |
| テスト戦略の組み込み | なし (都度指示) | なし | なし | **内蔵 (PBT / mutation / metamorphic)** |
| リファクタ指針の組み込み | なし | なし | なし | **内蔵 (CQRS / DDD / Pending Object)** |
| 別モデルによる交差レビュー | なし | なし | なし | **内蔵 (軍師ロール)** |

takumi は Claude Code の上に乗るスキルで、Claude Code 自体を置き換えるものではありません。**むしろ Claude Code の標準機能を前提に、その上で「長時間・大規模・高品質」を実現するための仕組みを積んでいる、と捉えてください。**

## 裏側でなにが起こっているか

気になる方向けに、takumi が自然文を受けてからの内部フローをご紹介します。

1. **意図分類** — 観点語 (security、パフォーマンス、a11y など) と診断動詞 (見て、調べて、心配) の組み合わせで、6 モードのどれに該当するかを判定します。曖昧なときだけ 1 問聞き返します。
2. **AC-ID 抽出** — ヒアリングでの発話から、要件を原子単位の受け入れ条件 (Acceptance Criteria) に分解し、`AC-AUTH-002` のような ID を付与します。**ユーザーが書く必要はありません。** 一覧を提示するので OK/修正を返すだけです。この ID が、仕様・計画・テスト・レビューをすべて紐付ける**共通言語**になります。
3. **計画生成と軍師レビュー** — 計画ファイルを書き出し、生成直後に軍師 (GPT-5) が敵対的にレビューします。矛盾・漏れ・リスクがあれば計画に反映してから、人間に提示します。
4. **Wave 実行** — ユーザーが承認したら、Wave 1 のタスクから順に職人 (Sonnet) が実装していきます。各タスクには検証ゲートがあり、通らなければ次の Wave に進みません。
5. **自己増殖** — 大きな総点検では、職人が実装中に見つけた別の問題が記録され、Wave 完了時に棟梁が計画に追記します。**計画そのものが成長していきます。**

## よくあるご質問

**Q. Claude Code 以外のハーネス (Cursor、Cline、Aider など) でも使えますか?**

現時点では Claude Code 専用です。takumi は Claude Code の skills システムの上に成り立っているため、同等の仕組みがない環境では動きません。将来的な移植可能性は検討中です。

**Q. 日本語以外でも動きますか?**

動きますが、意図分類の辞書が日本語に最適化されています。英語でも動作はしますが、観点語・診断動詞のマッチ精度はやや落ちます。`natural-language.md` に辞書を足すことで改善できます。

**Q. 途中で止めたら、書きかけのコードはどうなりますか?**

Wave 単位で中断されるため、完了済みの Wave の成果物は残り、途中だった Wave は破棄されます。次回 `/takumi 続きから` で、その Wave の冒頭から再開されます。

**Q. どこにファイルが書かれますか? 既存コードは勝手に触られませんか?**

takumi が書くファイルは `.takumi/` 配下のみです。既存コードは、ユーザーが明示的に「実装して」と頼んだときだけ変更されます。観点診断モードでは 1 行も触りません。

**Q. AC-ID とは何ですか? 覚えなくてはいけませんか?**

Acceptance Criteria (受け入れ条件) の ID です。たとえば `AC-AUTH-002` のような形をしています。要件を原子単位で表したもので、**仕様・計画・テスト・レビューをつなぐ共通言語**になります。ユーザーが書く必要はなく、AI が発話から抽出して提示するので、OK/修正を返すだけで大丈夫です。

**Q. mutation score の目標値はいくつですか?**

takumi のデフォルトは 65-80% の幅です。新規のドメインコードでは 80% 以上、既存レガシーに手を入れる場合は 65% など、プロジェクトの状況に応じて profile で可変です。`mutation_floor` を下回る場合は、次の Wave に進みません。

**Q. チームで共有すべきファイル、しないほうがよいファイルはどれですか?**

| ディレクトリ | 共有推奨 | 理由 |
|---|---|---|
| `plans/` | はい | レビュー対象になる |
| `specs/` | はい | 仕様 (AC-ID) そのもの |
| `design/` | はい | デザイン成果物 |
| `profiles/` | はい | チームの基準 |
| `sprints/` | いいえ | 個別セッションのログ |
| `telemetry/` | いいえ | 内部メトリクス |
| `control/` | いいえ | 一時的な指示 |

**Q. 実装中に予定外の問題を見つけたら、どうなりますか?**

担当外の発見は `.takumi/drafts/discovered-{id}.md` に記録され、その場では触られません。Wave 完了時に棟梁が統合して計画に追記し、次の Wave で扱います。重大な問題 (P0) は次バッチに割り込みで入ります。

**Q. TypeScript 以外の言語でも使えますか?**

使えますが、verify / strict-refactoring の主戦場は TypeScript / JavaScript です。Go / Rust / Python は `lang-relaxed-go-rust` profile で制約を緩めてあります (詳細は `strict-refactoring/language-relaxations.md`)。

**Q. 始めるにあたって必要な前提知識は?**

Claude Code を一度でも使ったことがあれば十分です。AC-ID や mutation score といった用語を事前に学ぶ必要はありません。takumi が対話のなかで必要なタイミングで説明します。

## インストール

gh CLI v2.90.0 以上が必要です。

```bash
gh skill install NAM-MAN/takumi
```

Claude Code を開いて、スラッシュコマンドの補完に `/takumi` が出てくれば成功です。

中身を事前に確認したい場合:

```bash
gh skill preview NAM-MAN/takumi takumi
```

アンインストール:

```bash
gh skill uninstall takumi
```

## `.takumi/` ディレクトリの中身

プロジェクト直下の `.takumi/` 以下だけにファイルが書かれます。

```
.takumi/
  plans/                        計画ファイル (Wave 構成)
  specs/                        AC-ID による仕様
  design/                       サイトマップ・スタイルガイド・ワイヤーフレーム
  profiles/                     verify / design / refactor の設定
  sprints/                      観点診断・総点検の実行記録
  telemetry/                    指標の時系列ログ
  control/                      一時停止などの制御ファイル
  state.json                    現在のモードと実行中 ID
  discovery-calibration.jsonl   発見者精度の履歴
```

`.gitignore` に追加を推奨するのは `sprints/` `control/` `telemetry/` です。残りはチームで共有したほうが効果的です。

## 制限事項

- Claude Code 専用です (Anthropic API を直接利用する環境では動きません)
- 意図分類は日本語に最適化されています
- 軍師ロール (交差レビュー) は OpenAI Codex CLI (GPT-5) を使用します。未インストール環境では Opus が代替しますが、精度はやや落ちます
- 自動判定を誤り続ける語彙があれば、`natural-language.md` の辞書に追加してください (PR 歓迎)
