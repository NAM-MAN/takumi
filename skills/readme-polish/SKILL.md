# readme-polish: README を GitHub プレビュー向けに磨き上げる

既存の README.md や markdown ドキュメントを、GitHub 上で読まれることを前提に、**日本の SaaS LP 風の読みやすさ**で書き直すスキル。

---

## When To Use

### 起動キーワード

- 「README 磨いて」「README を LP 風に」
- 「GitHub プレビューを整えて」「見た目を整えて」
- 「ドキュメントを読みやすく」「OSS っぽく整えて」
- 「バッジとかコールアウト入れて」「Mermaid 図入れて」

### 対象ファイル

- プロジェクト直下の `README.md`
- `skills/*/SKILL.md`、`docs/*.md`、`CONTRIBUTING.md` などの markdown
- 任意の md ファイル (引数で指定可)

単独起動でも、`/takumi` などから内部呼出でも可。

---

## 適用する 10 のパターン

入力の markdown を、以下のパターンで書き直す。**内容を削除・要約しない。並び替えと装飾・構造化のみ。**

### 1. バッジバー (中央寄せ)

タイトル直後に `<p align="center">` で中央寄せ。`<div>` ではなく `<p>` で統一。

```markdown
# skill-name: タグライン

<p align="center">
  <img alt="Skill" src="https://img.shields.io/badge/skill-<name>-7C3AED">
  <img alt="Status" src="https://img.shields.io/badge/status-<value>-success">
  <img alt="Meta" src="https://img.shields.io/badge/<key>-<value>-orange">
</p>
```

**色の定番**:
- 紫 `7C3AED` — スキル名・カテゴリ
- 緑 `success` — 肯定的属性 (stable、ready)
- オレンジ `orange` — 技術スタック・特徴

### 2. Hero タグラインと 1 ショットコマンド

タイトル下に **1 文の価値提案** + **代表的なコマンド** を置く。

```markdown
# skill-name: タグライン

...バッジ...

テストの質を、人間の経験ではなく機械で測る時代へ。

​```
/skill-name
​```

プロジェクトを診断し、〜を自動生成します。
```

### 3. アンカー付き目次 (長文のみ)

スクロール 2 画面以上の長文には必ず目次を入れる。

```markdown
## 目次

- [こんなお悩み、ありませんか?](#こんなお悩みありませんか)
- [〇〇 が解決すること (N 個の視点)](#〇〇-が解決すること-n-個の視点)
  - [1. サブセクション 1](#1-サブセクション-1)
- [用語解説 (初めて聞く方へ)](#用語解説-初めて聞く方へ)
```

> **アンカー生成規則**: 見出しをすべて小文字化、空白を `-` に、記号 (`?` `!` `(` `)` `?` `!` `(` `)`) を削除、かっこ内の文字は保持。

### 4. GitHub Alert でコールアウト

重要箇所に**色付きのコールアウト**を入れる。1 セクションに 1-2 個まで (乱用しない)。

```markdown
> [!NOTE]
> 補足情報。青色。

> [!TIP]
> 使いこなしのヒント。緑色。

> [!IMPORTANT]
> 重要な主張・核心。紫色。

> [!WARNING]
> 注意事項・既知の問題。黄色。

> [!CAUTION]
> 危険・破壊的挙動・絶対禁止。赤色。
```

**使い分けの指針**:

| Alert | 使いどころ |
|---|---|
| `NOTE` | 歴史的背景、補足、「ちなみに」の情報 |
| `TIP` | 推奨事項、お悩み解決のフック、使いこなし |
| `IMPORTANT` | スキルの核心主張、「これが一番大事」 |
| `WARNING` | 制限事項、カバレッジ至上主義の問題など |
| `CAUTION` | 禁止事項、データ消失リスク、著作権など |

### 5. Mermaid でフロー可視化

テキストで書かれたフロー・決定木・状態遷移を、Mermaid 図にする。**1 README に 1-3 枚まで**。

```markdown
​```mermaid
flowchart LR
    U[ユーザー] --> T{スキル}
    T -->|mode A| A[処理 A]
    T -->|mode B| B[処理 B]
​```
```

**図の種類の選び分け**:

| 種類 | 適する場面 |
|---|---|
| `flowchart LR` | 横方向の工程・パイプライン |
| `flowchart TD` | 縦方向の決定木 |
| `stateDiagram-v2` | 状態機械・ステータス遷移 |
| `sequenceDiagram` | アクター間のやり取り |

### 6. 用語解説テーブル (初学者向け配慮)

専門用語 (英略語、カタカナ語、技術名) をその場で解説するほか、**末尾に用語解説テーブル**を設ける。

```markdown
## 用語解説 (初めて聞く方へ)

| 用語 | 意味 |
|---|---|
| **PBT (Property-Based Testing)** | ランダム入力 1 万回で性質を検証する手法 |
| **mutation score** | 意図的に植えたバグをテストが検知した割合 |
```

### 7. `<details>` 折りたたみ

長い運用詳細・Agent 委譲プロンプト・サンプル出力は `<details>` で折りたたみ、**人間の読みやすさを優先**。ただし**中身は削除しない** (AI runtime で Read される可能性があるため)。

```markdown
<details>
<summary><b>Agent 委譲プロンプト (クリックで展開)</b></summary>

​```
Agent(
  description: "...",
  prompt: """..."""
)
​```

</details>
```

**折りたたみ候補**:
- 50 行以上のコードブロック
- 運用仕様の詳細 (Phase 定義、I/O 契約、禁止事項)
- 完了レポートのテンプレート
- 「採用前に決める閾値」などの細部
- 関連リソース一覧 (10 行以上)

### 8. テーブル崩壊の修正

セル内に `|` が混在すると markdown テーブルが崩壊する。

```markdown
<!-- NG: strictness の値にパイプが混在 -->
| 項目 | 値 |
|---|---|
| strictness | L1 | L2 | L3 |

<!-- OK: スラッシュ区切り -->
| 項目 | 値 |
|---|---|
| strictness | L1 / L2 / L3 |

<!-- OK: バッククォートでエスケープ -->
| 項目 | 値 |
|---|---|
| delimiter | `|` |
```

書き換え時は **全テーブル行を走査**し、セル内 `|` を修正する。

### 9. LP と技術仕様の明確な分離

人間向け LP と AI 向け運用仕様を 1 ファイルに共存させる場合、**h1 レベルの divider** で明確に分離する。

```markdown
人間向け LP パート
(お悩み、解決策、特徴、実例、FAQ、用語解説)

---

# 以下、AI 実行時に参照する仕様

> [!NOTE]
> ここから下は {スキル名} を実行する AI エージェントが読む仕様セクションです。人間が通読する必要はありません。

---

技術仕様 (Phase 定義、I/O 契約、制約など)
```

### 10. 比較テーブル (任意)

既存選択肢との違いを一覧化することで、**「なぜこれを選ぶか」を可視化**する。

```markdown
|  | 選択肢 A | 選択肢 B | **本スキル** |
|---|:-:|:-:|:-:|
| 軸 1 | - | ✓ | **✓** |
| 軸 2 | ✓ | ✓ | **✓** |
| 軸 3 | - | - | **✓ (独自)** |
```

中央寄せ (`:-:`) を使い、**本スキルの列だけ太字**で強調。

## スタイルルール

- **文体**: です・ます調 (日本語の場合)。LP ではトーンを丁寧に、技術仕様では簡潔に
- **トーン**: 日本の SaaS LP 風 (問題提起 → 解決策 → 特徴 → 実例 → FAQ)
- **専門用語**: 初出時に短く定義し、末尾に用語テーブルでまとめる
- **emoji**: 使わない (ユーザーから明示要求がない限り)
- **見出し階層**: h1 (タイトル) / h2 (セクション) / h3 (サブセクション) のみ。h4 以下は使わない
- **コードブロック**: 言語指定を必ず付ける (`​```bash`, `​```yaml`, `​```mermaid`)
- **リンク**: 相対パス優先 (`[text](path)`)
- **中央寄せ HTML**: `<p align="center">` で統一 (`<div>` ではなく)
- **太字**: 段落内の**強調する核心**のみ。多用すると効果が薄れる

## Workflow

1. **対象ファイル確認** — 引数または対話で対象 `README.md` (など) のパスを確定
2. **内容分析** — 現 README を Read し、以下を抽出:
   - 現行の章構成
   - 専門用語 (カタカナ語、英略語、技術名)
   - 長いコードブロック (50 行以上 → `<details>` 候補)
   - 崩壊テーブル (セル内 `|` 混在)
   - 既存のフロー図や表 (Mermaid 化候補)
   - LP パートと技術仕様の境界
3. **改善方針を提示** — 適用するパターン一覧を 5-10 行で簡潔に報告
4. **書き換え** — Write tool で全文を書き換え (Edit で段階的も可)
5. **検証** — Read で再読し、以下を確認:
   - リンク切れ (ファイル存在、アンカー)
   - markdown 構文エラー (Mermaid、テーブル、details タグ)
   - 内容の過不足 (元から失われた情報がないか)
6. **報告** — 適用した変更を箇条書きで 5-8 項目に要約

## 制約

> [!IMPORTANT]
> - **内容 (文字列) を削除・要約しない**。並び替えと装飾のみ許可
> - **frontmatter を変更しない** (skill ファイルの場合、`name` と `description` は保持)
> - **コードブロック・YAML・JSON の中身を変更しない** (bash、yaml、json など)
> - **Agent 委譲プロンプト (JSON スキーマ、I/O 契約、禁止事項) を改変しない**
> - **emoji は使わない** (ユーザーが明示要求した場合のみ)

> [!WARNING]
> - **Mermaid は過剰に入れない** (1 README に 1-3 枚が目安)
> - **GitHub Alert の乱用を避ける** (主張が薄れる、1 セクションに 1-2 個まで)
> - **`<details>` で必要な情報を隠しすぎない** (AI runtime で必要な仕様は残すが、人間の第一印象を損なわない配置に)

## 判断に迷ったとき

| 迷い | 判断指針 |
|---|---|
| Mermaid 化するか、テキストのままか | 3 分岐以上または状態遷移が明示的なら Mermaid |
| `<details>` で隠すか | 50 行超、またはサンプル/テンプレートなら隠す |
| Alert をどれにするか | 核心主張は `IMPORTANT`、禁止は `CAUTION`、注意は `WARNING` |
| 目次を入れるか | 2 スクロール以上 (h2 が 5 個以上) なら入れる |
| 比較テーブルを入れるか | 類似プロダクト/選択肢が 2 つ以上あるなら入れる |
| 用語テーブルを入れるか | 初学者向けの専門用語が 5 個以上あるなら入れる |
