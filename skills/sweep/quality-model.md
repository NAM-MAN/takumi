# Quality Model: MECE品質次元

## 8次元

| ID | 次元 | 焦点 | 発見者構成 |
|----|------|------|-----------|
| D1 | 機能正確性 | 仕様通りに動くか | バグハンター、エッジケース |
| D2 | UX | 使いやすいか | ペルソナ、UXデザイナー |
| D3 | 堅牢性 | 障害に耐えるか | エラーハンドリング、回復性 |
| D4 | 視覚一貫性 | 見た目が統一されているか | 横断パターン |
| D5 | セキュリティ | 安全か | 認証/認可、入力検証 |
| D6 | パフォーマンス | 速いか | N+1、バンドル、再レンダリング |
| D7 | アクセシビリティ | 誰でも使えるか | キーボード、aria、コントラスト |
| D8 | コード品質 | 保守しやすいか | アーキテクチャ、DX、死コード |

## 各次元の探索指示

### D1: 機能正確性
- **観点**: 状態遷移が仕様通りか、バリデーション漏れ、データ不整合
- **探索対象**: ドメインモデル、API route、状態遷移クラス

### D2: UX
- **観点**: 操作フィードバック、空状態、ローディング、導線の自然さ
- **探索対象**: UIコンポーネント、画面フロー、フォーム

### D3: 堅牢性
- **観点**: エラー表示、二重送信防止、楽観的更新ロールバック、オフライン
- **探索対象**: APIコール箇所、mutation、状態更新

### D4: 視覚一貫性
- **観点**: gap/padding/font統一、バッジ色、アクションボタン配置
- **探索対象**: 全画面横断でUIパターンを比較

### D5: セキュリティ
- **観点**: 認証/認可チェック、入力検証、秘密情報、RLS
- **探索対象**: auth/、API route、環境変数、middleware

### D6: パフォーマンス
- **観点**: N+1クエリ、不要な再レンダリング、バンドルサイズ、大量データ
- **探索対象**: DBクエリ、useEffect、import構成、リスト表示

### D7: アクセシビリティ
- **観点**: キーボード操作、aria-label、コントラスト比、フォーカス管理
- **探索対象**: インタラクティブUI、フォーム、モーダル

### D8: コード品質
- **観点**: 死コード、重複ロジック、型安全性、テスト容易性
- **探索対象**: 全体横断（構造分析）

## 優先順位ヒエラルキー（矛盾裁定基準）

```
Tier 1（絶対）: D5:セキュリティ ≥ D1:機能正確性
Tier 2（ユーザー価値）: D2:UX ≈ D7:アクセシビリティ ≈ D3:堅牢性
Tier 3（体験品質）: D4:視覚一貫性 ≈ D6:パフォーマンス
Tier 4（持続性）: D8:コード品質
```

- 同Tier内の矛盾: Synthesis必須 → 不可なら軍師裁定
- 異Tier間の矛盾: 上位Tier優先（ただしSynthesis可能なら常にSynthesis優先）

## 深度配分ルール

| 深度 | 発見件数指示 | 割当条件 |
|------|------------|---------|
| DEEP | 15-30件 | git diffで変更あり / 前回NG多数 |
| STANDARD | 10-20件 | デフォルト |
| SCAN | 最大10件 | 3回連続OK |
| SKIP | 0件 | 5回連続OK |

## 卒業メカニズム

`.takumi/quality-state.md` で画面×次元ごとに管理:
- 3回連続OK → SCAN降格
- 5回連続OK → SKIP
- 1回NG → STANDARD復帰
