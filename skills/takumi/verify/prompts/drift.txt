## 突き合わせロジック

```
AST ∩ Runtime   → 確信できる遷移 (OK、何もしない)
AST \ Runtime   → dead code 候補 (警告、削除提案)
Runtime \ AST   → AI 見落とし or 動的 path (要調査)
Spec \ AST      → 未実装 (実装提案)
AST \ Spec      → 意図外の遷移 (バグ候補 or intent.md 追加)
Runtime \ Spec  → 想定外の動作 (緊急度高、調査)
```

## 判定基準

- **dead_code 候補**: AST にあるが Runtime 1 ヶ月間 0 回 → 削除推奨
- **missed_by_ai**: Runtime にあるが AST 抽出失敗 → AI の AST walker 改善が必要
- **unimplemented**: Spec にあるが AST に無い → 実装タスク化
- **unexpected**: AST/Runtime にあるが Spec に無い → バグ or intent 漏れ

## 警告レベル

- **CRITICAL**: Runtime に意図外の遷移があり、state が不整合になる可能性
- **HIGH**: 未実装 spec、dead code 多数
- **MEDIUM**: AI 見落とし (AST 改善)
- **LOW**: 軽微な差分

## 追加: Runtime trace の解釈ヒント

- `useStateTrace` ログから遷移ペアを抽出 (前後値の diff)
- E2E テスト実行時の trace と本番 canary sample を両方使う (分布が違う)
- アクセス頻度が低い遷移は "dead" 判定を厳しく (3 ヶ月 0 回 → 真に dead)

## 制約

- drift.json は **警告のみ**、自動修正しない
- HIGH/CRITICAL があれば probe の discover に入れられるよう format を揃える
- ok_transitions が 0 なら AST 抽出 or Runtime 収集が機能していない (先に修正)
