# 選別フェーズ (probe mode の内部処理)

発見フェーズの出力 (discoveries.md) を入力とし、優先順位付きバックログを生成する。
この文書は takumi の probe mode Phase 2 で参照される内部手順書であり、単独スキルではない。

## 4ロール体制

| ロール | モデル | 担当 |
|--------|--------|------|
| 棟梁 | opus (自分) | 重複排除・MECE分類・ICE採点・統合 |
| 軍師 | gpt-5.4 (`codex exec`) | 反論者チェック |
| 斥候 | haiku (Agent tool) | 必要に応じた追加調査 |

## ファイル

| パス | 用途 |
|------|------|
| `.takumi/sprints/{日付}/discoveries.md` | 入力（発見フェーズの出力） |
| `.takumi/sprints/{日付}/backlog.md` | 出力（優先順位付きバックログ） |

---

## Step 1 — 重複排除

discoveries.md を読み、以下の基準で重複を統合:

1. **同一ファイル・同一問題**: 複数の発見者が同じファイルの同じ問題を指摘 → 1件に統合
2. **同一根本原因**: 異なるファイルだが原因が同じ（例: バリデーション方針の不統一） → 1件に統合、個別証拠は保持
3. **包含関係**: 大きな問題が小さな問題を含む → 大きな方に統合

統合時は「{N}名の発見者が指摘」と記録する。複数発見者の指摘 = 重要度の裏付け。

---

## Step 2 — MECE分類

重複排除後の課題を以下のカテゴリに分類:

```
機能要件
├── Bug（壊れている）
│   動くべきものが動かない、仕様と異なる挙動
├── UX（使いにくい）
│   動くが操作性・わかりやすさに問題がある
└── Missing（足りない）
    あるべき機能やフィードバックが存在しない

非機能要件
├── Performance（遅い）
│   N+1クエリ、不要な再レンダリング、バンドルサイズ
├── Security（危険）
│   認証/認可の不備、入力検証漏れ、秘密情報露出
├── Accessibility（使えない人がいる）
│   キーボード操作、スクリーンリーダー、コントラスト
├── Architecture（拡張しにくい）
│   高結合、責務の混在、パターン不統一
└── DX（開発しにくい）
    テスト困難、型安全性不足、ドキュメント不足
```

**ルール**: 1つの課題は1つのカテゴリに分類。迷った場合は影響の大きい方。

---

## Step 3 — ICE採点

各課題に3軸で1-5点を付与する:

### Impact（影響度）: ユーザーまたはシステムへの影響の大きさ

| 点数 | 基準 |
|------|------|
| 5 | 全ユーザーに影響 / データ損失リスク / セキュリティ脆弱性 |
| 4 | 多数のユーザーに影響 / 主要フローの阻害 |
| 3 | 一部のユーザーに影響 / 代替手段あり |
| 2 | 稀なケースで発生 / 見た目の問題 |
| 1 | 開発者のみが気づく / 極めて軽微 |

### Confidence（確信度）: 証拠の強さ

| 点数 | 基準 |
|------|------|
| 5 | 再現可能なバグ / コードに明確な欠陥 |
| 4 | コード上の問題が確認できる / 複数発見者が指摘 |
| 3 | 問題はあるが影響範囲が不明確 |
| 2 | 改善の余地はあるが現状でも動く |
| 1 | 推測ベース / ベストプラクティスとの乖離のみ |

### Ease（容易さ）: 修正の容易さ

| 点数 | 基準 |
|------|------|
| 5 | 1ファイル、数行の変更 |
| 4 | 2-3ファイル、テスト含めて1タスク |
| 3 | 複数ファイル、1Wave相当 |
| 2 | 設計変更を伴う、複数Wave相当 |
| 1 | アーキテクチャ変更、大規模リファクタ |

**ICEスコア = Impact × Confidence × Ease**（最大125、最小1）

---

## Step 4 — 反論者チェック

ICEスコア上位20件を 軍師 に渡し、反論を求める:

```bash
# 課題リストを1行1課題で変数に入れる（プロンプトに直接埋め込む）
codex exec -m gpt-5.4 -s read-only -C "$(pwd)" \
  "あなたは反論者（Devil's Advocate）である。
以下のプローブバックログについて、各課題に対して以下の観点で挑戦せよ:

1. YAGNI: 本当に今直す必要があるか？ユーザーが実際に困っている証拠は？
2. 代替案: もっと簡単な解決策はないか？
3. リスク: 修正による副作用（既存機能の破壊、依存関係の変更）は？
4. 順序: 他の課題を先に直すべきでは？（依存関係）

判定:
- ✅ 採用: 修正すべき（反論に耐えた）
- ⚠️ 要検討: 価値はあるが条件付き
- ❌ 却下: 今は不要（理由を明記）

各課題について判定と理由を日本語で述べよ（1課題あたり3-5行）。

プローブテーマ: {テーマの1行説明}

課題リスト:
B-001. {タイトル} [ICE:{スコア}] - {証拠の要約}
B-002. {タイトル} [ICE:{スコア}] - {証拠の要約}
..." 2>&1 | tail -100
```

**注意点:**
- `-o` オプションは存在しない。結果は stdout に出るので `2>&1 | tail -100` でノイズ除去
- プロンプトは1つの文字列引数として渡す（ヒアドキュメント不要）
- 課題リストはプロンプト内に直接展開する（外部ファイル参照ではなく）
- MCP起動ログが stderr に出るが `2>&1 | tail` で無視される
- timeout 120秒を設定する（Bash tool の timeout パラメータ）

軍師 の判定結果を反映:
- ✅ 採用 → バックログに含める
- ⚠️ 要検討 → ICEスコアを0.7倍に調整してバックログに含める
- ❌ 却下 → バックログから除外（却下理由を記録）

---

## Step 5 — プローブ選出

### 選出基準

1. 反論者チェック後のICEスコアで降順ソート
2. 上位 **8-15件** を選出（1プローブの適正量）
3. 依存関係がある場合はセットで含める

### 依存関係の判定

- 同じファイルを修正する課題 → 同一プローブに含める
- Aを直さないとBが直せない → Aを含めるならBも含める
- アーキテクチャ変更 → それに依存する課題も含める

```
→ 計画フェーズに進みます。
```

---

## 制約

- discoveries.md がなければエラー終了（先に発見フェーズを実行）
- ICE採点は 棟梁 自身が行う（サブエージェントに委譲しない）
- 反論者チェックは必ず 軍師 に委譲（自分で反論しない）
- 0件選出はあり得る（全て却下された場合、ユーザーに報告して終了）
